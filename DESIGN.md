## Problem Statement

中文音乐学习者缺乏一个统一的、设计良好的、交互式的乐理学习网站。当前的学习路径是碎片化的：B站视频、知乎文章、英文网站（musictheory.net、teoria.com）。没有一个中文优先的站点能同时提供结构化的知识体系和交互式的音频/视觉学习体验。

现有解决方案的问题：
- **musictheory.net** — 交互好但英文、线性、UI 过时
- **Musicca** — 有中文但交互浅、UI 通用
- **B站/知乎** — 中文但纯视频/文本、无交互
- **muted.io** — 工具精美但无学习路径
- **音壳** — 中文但以 App 为主，不是 Web 知识库

## What Makes This Cool

一个打开即用的乐理知识库：不卖课、不需注册、不强制线性学习。像 MDN 之于 Web 开发，Music Base 之于音乐理论 — 你想学什么就点什么，每个概念都有交互式钢琴可以看到、听到。中文互联网上第一个这样的东西。

## Constraints

- **中文优先**：所有内容以中文为主，英文术语作为辅助标注
- **多端可访问**：桌面端和移动端均可使用，响应式设计
- **零门槛**：不需要注册、不需要付费、不需要安装任何东西
- **Web 技术栈**：Web Audio API + Canvas/SVG 足以实现所有交互
- **v1 以钢琴为主**：多乐器和 MIDI 支持是 v2 目标

## Premises

1. **市场空白真实存在** — 中文学习者目前只能拼凑 B站/知乎/英文网站，没有统一的交互式中文乐理资源
2. **非线性知识库 > 线性课程** — MDN 式导航，用户选择自己的学习路径，不是步步为营的课程
3. **Web 足够** — Web Audio API + Canvas/SVG 可以处理钢琴可视化和音频播放，不需要原生应用
4. **小而完整地起步** — 课程地图 + 几个完整的交互式主题，比试图覆盖所有乐理知识更好
5. **钢琴优先，多乐器后做** — v1 以钢琴作为主要可视化工具，吉他/MIDI 是 v2

## Approaches Considered

### Approach A: VitePress + Vue Components ✅ (Chosen)
- **Summary:** VitePress（Vue 官方文档框架）+ Markdown 内容 + 嵌入式 Vue 交互组件
- **Effort:** S | **Risk:** Low
- **Pros:**
  - 为知识库场景量身定制：内置搜索、侧边栏导航、移动端响应
  - 内容用 Markdown 写，开发体验极佳
  - 内置 i18n 支持，未来国际化成本低
  - 部署简单（Vercel/Netlify/GitHub Pages）
- **Cons:**
  - 未来加复杂状态（用户进度、MIDI）会稍难
  - Vue 生态的音乐库比 React 少
- **Reuses:** VitePress 内置导航、搜索、响应式布局

### Approach B: Next.js + MDX + React
- **Summary:** Next.js App Router + MDX 内容页 + React 交互组件库
- **Effort:** M | **Risk:** Low
- **Pros:** 最好的长期灵活性，React 生态最大，容易加功能
- **Cons:** 对于主要是静态的内容来说太重了，更多模板代码

### Approach C: Astro + Preact Islands
- **Summary:** Astro 静态内容 + Preact 岛屿组件
- **Effort:** M | **Risk:** Med
- **Pros:** 最佳性能（对中国移动端用户重要），最小的 JS 包体积
- **Cons:** 生态较小，岛屿模式有学习曲线，i18n 不成熟

## Recommended Approach

**VitePress + Vue Components** — 因为它出厂即是知识库，内置搜索/导航/i18n/移动端适配。交互组件（钢琴、音频）用的是框架无关的 Web Audio API + Canvas，所以核心工作不会因为框架选择而浪费。以最快速度交付一个可用的产品，需要时再迁移。

### 交互组件技术方案（由 /plan-eng-review 确认）

- **music.ts**: 共享数据模型 — Note/Interval/Scale/Chord 类型 + noteToMidi/midiToFrequency/getInterval/getScaleNotes/getChordNotes 工具函数。所有组件通过此模块共享音乐理论逻辑，避免 DRY 违规
- **useAudio.ts**: Vue composable — 管理 Tone.start() 生命周期（首次用户交互时初始化），提供共享 Sampler 单例。若 AudioContext 处于 suspended 状态，显示"点击启用音频 🔊"提示
- **Piano.vue**: SVG 渲染钢琴键盘。Props: `highlighted: string[]`, `root?: string`, `range: [number, number]`（默认 [48,72] = C3-C5）, `interactive: boolean`（默认 true）。Events: `@note-click(note: string)`。所有交互组件在 Markdown 中使用 `<ClientOnly>` 包裹以避免 SSR 崩溃
- **音频方案**: 采用 Tone.js + tonejs-instruments（Salamander Grand Piano 采样）。采样文件按需懒加载（用户首次点击钢琴时才加载），加载完成前使用 OscillatorNode 合成音作为即时回退，确保零等待交互
- **CJK 搜索**: 使用 VitePress 内置 MiniSearch + Intl.Segmenter('zh-CN') 自定义分词器，无需 Algolia
- **测试**: Vitest（单元/组件测试）+ Playwright（E2E 测试）
- **响应式**: 钢琴组件在移动端自动缩放，支持触屏点击

### 非线性导航设计

- 首页是 **课程地图**（非课程列表）：9 个主题卡片以网格形式展示
- 每个卡片标注主题名、英文名、节数
- "推荐起点" 标记建议初学者从哪里开始，但不强制
- 每个主题详情页底部有 **相关主题** 链接，形成知识网络
- 侧边栏内展示当前主题的章节列表 + 相关主题跳转
- 全局搜索支持中文/英文术语（注意：VitePress 内置 MiniSearch 对 CJK 分词支持有限，需配置自定义分词器或使用 Algolia DocSearch 的中文支持）

## Open Questions

1. **部署目标**: Vercel（推荐，零配置）、Netlify、还是 GitHub Pages？
2. **域名**: 是否已有域名？会影响 SEO 和分享
3. **内容编写**: 乐理内容谁来写？用户自己写还是 AI 辅助生成初稿后人工审核？

## Success Criteria

- 一个可访问的网站，桌面端和移动端均可正常使用
- 首页课程地图可以非线性导航到任何主题
- 至少 3 个主题有完整的交互组件（钢琴可视化 + 音频播放）
- 全局搜索可以用中文和英文搜索内容（使用自定义 CJK 分词器或 Algolia DocSearch）
- 用户无需注册即可使用所有功能
- 页面加载速度 < 3 秒（移动端 4G 网络）— 钢琴采样懒加载，不计入首屏时间

## Distribution Plan

- **部署**: Vercel（推荐）或 Netlify，从 GitHub main 分支自动部署
- **CI/CD**: GitHub Actions — push 到 main 时自动构建并部署
- **分发渠道**: 直接分享 URL，在知乎/小红书/B站等中文平台推广
- **开源**: GitHub 公开仓库，接受社区贡献内容

## What I noticed about how you think

- 你说"我仅仅是想要快速学习、做一个开箱即用的知识库网站" — 这种"解决自己的问题然后分享给别人"的思路，是最好的产品直觉。你不是在猜测别人需要什么，而是在做自己需要的东西。
- 你选了"Full curriculum overview"作为起点，而不是直接跳到某个具体主题 — 说明你重视结构化思维，想先建立全局观再深入细节。这和你选择非线性导航是一致的：你信任学习者自己选择路径的能力。
- 你选了"中文优先"而不是"先做英文再翻译" — 这是对目标用户的精准理解。中文乐理资源确实分散且质量参差不齐，你在为一个真实的空白做事。
- 你的 10x 愿景是"Multi-instrument + MIDI"，不是"加社区/加AI" — 这说明你关注的是工具本身的深度，而不是追热点。
