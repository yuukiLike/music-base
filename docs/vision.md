# Vision

> 这篇文档记录 Music Base 的长期思考，不是待办事项。

## 本质：重新定义书籍

传统书籍把知识**序列化成文字**。读者再把文字**反序列化成理解**。这个过程是有损的。

```
作者的理解 → [序列化为文字] → 文字 → [反序列化] → 读者的理解
                  ↑ 损耗                    ↑ 损耗
```

我们做的是压缩这个损耗：把知识编码成**结构化数据 + 交互体验**，读者在体验中直接获得理解。

```
作者的理解 → [编码为数据+交互] → 体验 → 读者的理解
                                  ↑ 接近无损
```

| | 传统书籍 | 新载体 |
|--|--|--|
| 编码 | 文字（有损压缩） | 文字 + 数据 + 交互（接近无损） |
| 阅读 | 被动解码 | 主动体验 |
| 感官 | 视觉 | 视觉 + 听觉 + 操作 |
| 结构 | 线性 | 图谱，多入口 |
| 载体 | 固定 | 同一份知识，多种投影 |
| 迭代 | 再版 | 持续演进 |
| 边界 | 单领域 | 概念图谱跨领域连接 |

这不是电子书，不是在线课程，是一种**新的知识载体**。

## 技术验证

Music Base 验证了一个模式：

```
知识库 = Markdown 内容 + 领域数据模型 + 交互组件包
```

Markdown 负责叙事，数据模型负责结构化知识，交互组件让概念可触摸。三者通过 VitePress 组合。

这个模式是**领域无关**的。

## 三层扩展

### Layer 1：多渲染层

同一份结构化知识，喂给不同的前端：

| 渲染层 | 形态 | 驱动力 |
|--------|------|--------|
| Web (VitePress) | 文章 + 交互组件 | 好奇心驱动，自由浏览 |
| App (多邻国模式) | 闯关 + 反馈 | 游戏化驱动，系统推题 |

关键：不是做两个产品，是**同一个知识库的两个投影**。

内容单元拆到足够原子化（一个概念 = 一条结构化数据），就能同时生成文章段落和练习题目。

### Layer 2：跨领域复制

把音乐替换成任何领域，架构不变：

- **物理**：力学模拟器、波形可视化
- **化学**：分子构建器、反应动画
- **编程**：在线执行器、AST 可视化
- **设计**：色彩交互、排版实验
- **烹饪**：温度曲线、时间线编排

变的是组件库和数据模型，不变的是 `Markdown + 数据 + 组件` 的组合方式。

Music Base 是这个框架的第一个实例。

### Layer 3：跨领域概念映射

不同领域的术语之间存在深层类比：

| 音乐 | 物理 | 编程 | 设计 |
|------|------|------|------|
| 频率 | 波长 | 循环周期 | — |
| 和声 | 共振 | 组合模式 | 构图 |
| 节奏 | 周期运动 | 事件循环 | 节奏感 |
| 泛音 | 傅里叶分解 | 分层抽象 | 层级 |
| 调性 | 参考系 | 命名空间 | 设计系统 |
| 休止 | 真空 | 空操作 | 留白 |

这构成一个**概念图谱**：节点是概念，边是跨领域类比。

价值：**让专业人士通过自己熟悉的领域，理解陌生领域。** 程序员通过"事件循环"秒懂"节奏"，音乐人通过"和声"理解"组合模式"。

## 不做什么

- **不做组件通信协议** — 组件间状态管理是已解决的工程问题（Vue provide/inject、Pinia），不需要再造轮子
- **不做通用框架** — 框架是从多个成功实例中提炼的，不是先设计出来的
- **不做基础设施** — 用户不在乎 Piano 怎么把数据传给 PianoRoll，用户在乎弹一个音能同时看到什么

## 路线

先做出让人惊叹的内容，再从内容中提炼模式。

1. **把音乐这本"书"做到极致** — 听音训练、和弦进行、五度圈、波形可视化
2. **做第二本"书"** — 选一个新领域，用同样的 Markdown + 数据 + 组件方式，验证模式可复制
3. **从两本书中提炼共性** — 此时框架自然浮现，而不是凭空设计
